Description

In the previous stage, you have created a traditional Django page that displays the tasks added by the superuser. In this stage, you will implement the same functionality but as an API.

Like in the previous stage, your model should include the title, description, two fields with dates, and a boolean field. The functionality of these fields is the same as in the previous stage. We will continue to improve our project in the following stages, so writing a new model is encouraged. Also, add some tasks in this app with the superuser created in the previous stage.

Start by implementing two simple GET endpoints: GET /api/tasks and GET /api/tasks/<id>. These endpoints will work similarly to the URL routing. While displaying the data, you need to order the tasks by the status of completion (incomplete first), by the due date (sooner ones first), and by the date of creation (the older ones first), in this order.

In the following stages, we will work on the POST methods as well. Previously, we've sealed with generic view classes of Django; now, you need to use the classes defined in the generics module of the rest_framework, ListAPIView, and RetrieveAPIView. You can import these classes using the following command: from rest_framework import generics. If you don't know much about them, take a look at the official documentation.

To complete this stage, you might need basic knowledge of the way class-based views, function-based views and generic views are used in API.

Objectives

Implement two endpoints:

  • GET /api/tasks returns all the tasks the user has added as a JSON object.

  • GET /api/tasks/<id> returns the detailed view of the task as per the task's ID.
